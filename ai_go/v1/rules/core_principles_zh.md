# 核心原则

AI 在每次开发交互中**必须**遵循以下原则。

## 不可妥协的必须项

- **禁止揣测**：遇到疑惑、不明确的需求或问题时，**禁止**自行揣测，**必须**向用户询问澄清。
- **确认优先**：在不确定用户意图时，**先确认再执行**。
- **YAGNI**（You Aren't Gonna Need It）：只实现**明确要求**的功能，**不得**添加未 requested 的扩展或特性。
- **变动必记**：所有变动—无论新特性或 bug 修复—**必须**记录到项目根目录下的 `docs/` 目录。
- **历史代码禁止删除**：**不得**删除历史代码，只能**注销**（注释）替代。
- **征得同意再改**：**禁止**修改原有代码的类型、入参、业务逻辑等内容；除非必须修改时，**必须**先征得用户同意。

## 原因说明

| 原则 | 原因 |
|-----------|-----|
| 禁止揣测 | 揣测会导致错误方案和无效工作。澄清可节省时间。 |
| 确认优先 | 避免因预期不一致而产生的返工。 |
| YAGNI | 保持范围聚焦，降低复杂度，尊重用户控制权。 |
| 变动必记 | 保持可追溯性与团队对齐。 |
| 历史代码禁止删除 | 保留审计轨迹，便于回滚。 |
| 征得同意再改 | 避免破坏调用方与意外副作用。 |

## 示例

**不良**：用户说「加个校验」→ AI 为 5 个字段加了全套校验，而用户只想校验邮箱格式。

**良好**：用户说「加个校验」→ AI 追问：「需要校验哪些字段？约束是什么（格式、长度、必填）？」

**不良**：用户说「重构一下」→ AI 未确认范围就直接改了结构。

**良好**：用户说「重构一下」→ AI 追问：「具体想改进哪些方面—抽取函数、重命名、拆分包，还是其他？」

**不良**：替换函数实现时直接删除旧代码。

**良好**：将旧代码注释掉，在其下方或邻近处添加新实现，并在 `docs/` 中记录变动。

**不良**：未征求用户同意就修改函数签名或返回类型。

**良好**：「修改会影响调用方 A、B，是否同意？我将在 docs/ 中记录。」
